# MoguMap ロードマップ

## 1. この文書の目的

この文書では、MoguMap開発の現在地、工程の順序、保留事項だけを管理する。

個別のコマンドや一時的な確認結果は記録しない。

具体的な要件は`MoguMap-requirements.md`、技術構成は`MoguMap-technical-design.md`、決定理由は`MoguMap-decisions.md`で管理する。

## 2. 現在地

- 現在の工程：工程1「検索画面の土台を作る」
- 状態：進行中
- 現在のブランチ：`feature/search-form`
- 次の区切り：検索フォームと関係文書を確認し、Pull Requestを通してmainへ反映する

現在、次の検索フォームを実装している。

- 千葉県内の中エリアを選択できる
- 中エリアの選択を必須とする
- 任意のキーワードを入力できる
- 検索条件をURLへ入れられる
- URLにある検索条件をフォームへ戻せる

ホットペッパーグルメAPIとの通信と、検索結果の表示はまだ実装していない。

## 3. 方針を決めるタイミング

未決定事項を最初にすべて決める工程は設けない。

各工程で実物のコードや動作を確認し、その工程に必要な内容だけを決定する。

今後の設計に影響する決定は`MoguMap-decisions.md`へ記録し、関係する要件定義書と技術設計書も必要な範囲だけ更新する。

## 4. 開発工程

| 工程                          | 状態   | 目標                                                    |
| ----------------------------- | ------ | ------------------------------------------------------- |
| 工程0：プロジェクト文書の整備 | 完了   | 4つの管理文書を確認し、mainへ反映する                   |
| 工程1：検索画面の土台を作る   | 進行中 | 中エリアとキーワードを入力し、URLへ保持できるようにする |
| 工程2：サーバー側APIを作る    | 未着手 | APIキーを隠したままホットペッパーグルメAPIと通信する    |
| 工程3：検索結果一覧を作る     | 未着手 | 店舗を1ページ10件ずつ表示する                           |
| 工程4：店舗詳細ページを作る   | 未着手 | 選んだ店舗の詳細情報を表示する                          |
| 工程5：最初の完成確認を行う   | 未着手 | 必須機能、失敗時の動作、品質、安全性を確認する          |

## 5. 保留事項

- サーバー側APIにNext.jsのRoute HandlerとExpressのどちらを使うか
- 検索結果一覧と店舗詳細ページのURL
- ページ番号をURLへ持たせる方法
- 店舗詳細ページに表示する具体的な項目
- 検索結果が0件だった場合の表示
- 外部APIでエラーが起きた場合の表示
- データベース
- 認証
- Docker
- AWS
- 対象地域の全国への拡大

各項目は、関係する工程へ進んだときに実物を確認しながら決める。

## 6. 更新ルール

ロードマップは、次の場合だけ更新する。

- 現在の工程が変わったとき
- 工程が完了したとき
- 工程の順序や目標が変わったとき
- 保留事項を実施すると決めたとき

個別のコマンド実行、lint、build、一時的なデバッグは記録しない。

工程の区切りでは、アシスタントが更新の必要性を確認し、必要な場合は具体的な変更を案内する。

新しいチャットでは、この文書の現在地を確認してから開発を続ける。
