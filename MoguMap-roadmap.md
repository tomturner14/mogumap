# MoguMap ロードマップ

## 1. この文書の目的

この文書では、MoguMap開発の現在地、工程の順序、保留事項だけを管理する。

個別のコマンドや一時的な確認結果は記録しない。

具体的な要件は`MoguMap-requirements.md`、技術構成は`MoguMap-technical-design.md`、決定理由は`MoguMap-decisions.md`で管理する。

## 2. 現在地

- 現在の工程：工程0「プロジェクト文書の整備」
- 状態：進行中
- 現在のブランチ：`feature/project-documents`
- 次の区切り：4文書を確認し、Pull Requestを通してmainへ反映する

検索画面とホットペッパーグルメAPIとの通信は、まだ実装していない。

## 3. 開発工程

| 工程                          | 状態   | 目標                                                   |
| ----------------------------- | ------ | ------------------------------------------------------ |
| 工程0：プロジェクト文書の整備 | 進行中 | 4つの管理文書を確認し、mainへ反映する                  |
| 工程1：最初の実装方針を決める | 未着手 | サーバー側API、URL、検索条件、未確定の表示仕様を決める |
| 工程2：検索画面の土台を作る   | 未着手 | 中エリアとキーワードを入力できるようにする             |
| 工程3：サーバー側APIを作る    | 未着手 | APIキーを隠したままホットペッパーグルメAPIと通信する   |
| 工程4：検索結果一覧を作る     | 未着手 | 店舗を1ページ10件ずつ表示する                          |
| 工程5：店舗詳細ページを作る   | 未着手 | 選んだ店舗の詳細情報を表示する                         |
| 工程6：最初の完成確認を行う   | 未着手 | 必須機能、失敗時の動作、品質、安全性を確認する         |

## 4. 保留事項

- データベース
- 認証
- Docker
- AWS
- 対象地域の全国への拡大

採用すると決めた場合だけ、開発工程へ追加する。

## 5. 更新ルール

ロードマップは、次の場合だけ更新する。

- 現在の工程が変わったとき
- 工程が完了したとき
- 工程の順序や目標が変わったとき
- 保留事項を実施すると決めたとき

個別のコマンド実行、lint、build、一時的なデバッグは記録しない。

工程の区切りでは、アシスタントが更新の必要性を確認し、必要な場合は具体的な変更を案内する。

新しいチャットでは、この文書の現在地を確認してから開発を続ける。

## 6. 次の一手

4つの管理文書をPull Requestを通してmainへ反映し、工程0を完了する。
